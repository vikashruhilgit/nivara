## Task
**Goal:** Prepare InvestIQ for beta launch with 10-20 users on Alpaca paper trading. EAS Build profiles (dev, preview, production) in eas.json. TestFlight (iOS) + Play Internal Testing (Android) distribution. Beta onboarding flow (sign up, select base currency, connect Alpaca paper account, first sync, land on portfolio dashboard). Enhanced health check endpoints (/health, /health/brokers, /health/data). Error tracking via Sentry free tier (backend + mobile) + Expo crash reporting. Monitoring alerts: API error rate thresholds with auto kill switch, broker sync failure alerts.

**Problem Statement:**
All features are built (Jobs 1-23) but the app is not deployable to real beta testers. There is no build pipeline for app store distribution, no onboarding flow for new users, no health monitoring for the production environment, and no error tracking to catch issues during beta. Without this job, the app exists only in local development. Beta launch is the critical milestone that validates the entire MVP with real users.

## Acceptance Criteria
- [ ] Given `eas build --profile production --platform ios`, then builds successfully with correct bundle identifier and signing
- [ ] Given `eas build --profile production --platform android`, then builds successfully with correct package name
- [ ] Given TestFlight build uploaded, then beta tester can install app and reach sign-up screen
- [ ] Given new user on first app launch, when onboarding starts, then flow is: sign up -> select base currency (INR/USD) -> connect Alpaca paper account -> first sync -> lands on portfolio dashboard
- [ ] Given GET /health, then returns JSON with DB connection status + Redis connection status + Celery worker status
- [ ] Given GET /health/brokers, then returns per-broker connection status, token freshness (fresh/expired), and feature flags
- [ ] Given GET /health/data, then returns pipeline freshness per data source (Yahoo Finance, GNews, FRED, FinBERT) with stale flags
- [ ] Given API error rate >5% over 5-minute window, then warning email sent to admin
- [ ] Given API error rate >15% over 5-minute window, then kill switch auto-activated + admin notified via Sentry alert
- [ ] Given broker sync fails 3 consecutive times for a user, then user receives in-app notification with troubleshooting action
- [ ] Given unhandled exception in mobile app, then reported to Sentry with device info, OS version, and user context
- [ ] Given unhandled exception in backend, then reported to Sentry with request context, user_id, and stack trace

## Risk Assessment

| Risk | Impact | Mitigation |
|------|--------|------------|
| Apple provisioning profile / signing issues for TestFlight | HIGH | Use EAS managed signing (--auto-submit); create App Store Connect app ID in advance; test with preview profile first |
| Sentry free tier limits (5K errors/month) may be exceeded during beta | LOW | 10-20 beta users unlikely to generate 5K errors; add sampling rate 1.0 for beta, reduce for production |
| Kill switch auto-activation false positive (brief API spike triggers it) | HIGH | Use 5-minute sliding window (not instantaneous); require sustained >15% error rate; add manual override endpoint |
| Onboarding flow interruption (user kills app mid-flow) | MEDIUM | Persist onboarding state in expo-secure-store; resume from last completed step on next launch |
| Health check endpoints expose internal infrastructure details | LOW | /health endpoints behind admin auth or internal-only network in production; return only status booleans + timestamps publicly |
| EAS Build requires Expo account + paid plan for production builds | MEDIUM | Use EAS free tier (limited builds/month); ensure Expo account is set up before starting |
| TestFlight review delay (Apple review can take 24-48h for first build) | MEDIUM | Submit TestFlight build early; use internal testing group (no review needed for first 100 testers) |
